# Рабочий процесс: маленькие шаги и откат

## Версии на GitHub и откат

- **Все запушенные коммиты** хранятся на GitHub. Каждый `git push` отправляет туда новые коммиты — они не пропадают.
- **Откатиться к любой версии можно:** по истории коммитов (локально `git reset` или через клонирование старого состояния с GitHub).
- **Как понять, какая версия что из себя представляет:** по сообщениям коммитов и по тегам (см. ниже). Поэтому важно **подписывать коммиты** — писать, что сделано и в каком состоянии проект.

## Как подписывать коммиты (чтобы потом понимать версии)

Сообщение коммита — это и есть «подпись» версии. Пишите **что сделано** и при желании **работоспособность**.

**Примеры хороших сообщений:**

```bash
git commit -m "Шапка и футер на главной, страница открывается"
git commit -m "Каталог: сетка карточек, ссылки на Озон/WB работают"
git commit -m "[рабочий] Главная и каталог готовы, можно пускать рекламу"
git commit -m "Исправлена вёрстка на мобильных — всё стабильно"
```

**Краткий формат, которого можно придерживаться:**

| Что писать | Пример |
|------------|--------|
| Что сделано | «Добавлена шапка», «Подключён tokens.css» |
| Где | «на главной», «в каталоге» |
| Работоспособность (по желанию) | «работает», «всё открывается», «[рабочий]» |

Тогда в истории (`git log`) или на GitHub во вкладке **Commits** будет видно: эта версия — «шапка и футер», эта — «каталог готов», эта — «последние правки мобильной вёрстки».

**Особо важные версии (релизы) можно помечать тегом:**

```bash
# Пометить текущий коммит как «версия 1 — витрина готова»
git tag -a v1-vitrina -m "Главная + каталог, можно пускать рекламу"

# Отправить теги на GitHub
git push origin v1-vitrina
```

На GitHub во вкладке **Releases** (или в списке тегов) потом будет видно: например, `v1-vitrina` — стабильная витрина. К любой такой версии можно откатиться по тегу: `git checkout v1-vitrina`.

## Правило: один логический шаг = один коммит

- Добавили один блок на страницу — коммит.
- Поправили стили в одном файле — коммит.
- Добавили новый файл (страницу, компонент) — коммит.

Так проще откатиться: «откатить последний коммит» отменяет только последнее изменение.

## Как коммитить

```bash
# Посмотреть, что изменилось
git status

# Добавить только нужные файлы (или все)
git add путь/к/файлу
# или
git add .

# Коммит с понятным сообщением
git commit -m "Добавлен блок героя на главную"
```

Сообщение коммита лучше писать по-русски и конкретно: что сделано, а не «фиксы».

## Как откатиться на предыдущий коммит

**Вариант 1 — отменить последний коммит, но оставить изменения в файлах (можно доработать и закоммитить снова):**
```bash
git reset --soft HEAD~1
```

**Вариант 2 — отменить последний коммит и убрать изменения в файлах (как до коммита):**
```bash
git reset --hard HEAD~1
```
⚠️ Изменения после `--hard` не восстановить. Используйте, только если уверены.

**Вариант 3 — откатиться на конкретный старый коммит (посмотреть историю):**
```bash
git log --oneline
git reset --hard <хеш-коммита>
```

**Вариант 4 — откатиться к версии по тегу (если помечали релизы):**
```bash
git tag -l
git checkout v1-vitrina
```

**Как посмотреть историю и найти нужную версию:**
```bash
git log --oneline          # краткий список коммитов
git log --oneline -20      # последние 20
```
На GitHub: репозиторий → вкладка **Commits** — там все запушенные версии и их сообщения.

## Перед началом вёрстки

1. Убедитесь, что репозиторий инициализирован: в корне есть папка `.git`.
2. Сделайте первый коммит со всей подготовкой (структура, план, документы), чтобы была «точка отката».

## Рекомендация

Делайте коммит после каждой законченной маленькой задачи. Если что-то сломалось — откат будет безболезненным.
